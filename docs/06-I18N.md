# 06-I18N.md

# Internationalization (i18n) – MLMS

## Ziel

Dieses Dokument definiert die vollständige Internationalisierungs-Strategie für das MLMS.

Das System ist für Marokko vorgesehen und muss deshalb von Anfang an **zweisprachig** aufgebaut werden:

- Französisch (`fr`)
- Arabisch (`ar`)

Arabisch muss zusätzlich vollständig **RTL** (Right-to-Left) unterstützt werden.

Dieses Dokument ist die verbindliche Grundlage für:

- Frontend Texte
- RTL Verhalten
- Übersetzungsdateien
- medizinische Begriffe
- Rollen- und Menütexte
- Formulartexte
- Status-Texte
- Report-Texte
- Patientenportal-Texte

---

## Grundprinzipien

### 1. Keine harten Texte in Komponenten
Alle sichtbaren Texte müssen aus Translation-Dateien kommen.

### 2. Ein Key – zwei Sprachen
Jeder Translation-Key muss in `fr` und `ar` existieren.

### 3. Arabisch ist echtes RTL
Arabisch ist nicht nur eine Übersetzung, sondern benötigt eine echte Layout-Spiegelung.

### 4. Fachbegriffe konsistent halten
Medizinische Begriffe dürfen nicht jedes Mal anders übersetzt werden.

### 5. Fallback Sprache ist Französisch
Wenn eine Übersetzung fehlt, wird `fr` verwendet.

---

## Unterstützte Sprachen

```text
fr = Français
ar = العربية
```

## RTL Konfiguration

## Datei: `frontend/src/i18n/rtl.ts`

```ts
export function applyDirection(language: 'fr' | 'ar') {
  const dir = language === 'ar' ? 'rtl' : 'ltr';

  document.documentElement.lang = language;
  document.documentElement.dir = dir;
  document.body.dir = dir;

  return dir;
}
```

## Datei: `frontend/src/i18n/helpers/getDirection.ts`

export function getLanguageLabel(language: 'fr' | 'ar'): string {

  return 'Français';
}

## Normalize Language Helper

## Datei: `frontend/src/i18n/helpers/normalizeLanguage.ts`

```ts
export function normalizeLanguage(input?: string | null): 'fr' | 'ar' {
  if (!input) return 'fr';

  const normalized = input.toLowerCase();

  if (normalized === 'ar') return 'ar';
  if (normalized === 'fr') return 'fr';

  return 'fr';
}
```

## Provider Integration

## Beispiel für `I18nProvider.tsx`

```tsx
import { useEffect } from 'react';
import { useTranslation } from 'react-i18next';
import { applyDirection } from '@/i18n/rtl';

type Props = {
  children: React.ReactNode;
};

export function I18nProvider({ children }: Props) {
  const { i18n } = useTranslation();

  useEffect(() => {
    applyDirection(i18n.language as 'fr' | 'ar');
  }, [i18n.language]);

  return <>{children}</>;
}
```

## Sprachwechsel – Regeln

Wenn Sprache gewechselt wird, muss das Frontend:

1. `i18n.changeLanguage(...)` aufrufen
2. `document.documentElement.lang` aktualisieren
3. `document.documentElement.dir` aktualisieren
4. User-Präferenz optional per API speichern
5. Layout neu rendern

## Sprachumschalter – Beispielkomponente

```tsx
import { useTranslation } from 'react-i18next';

export function LanguageSwitcher() {
  const { i18n } = useTranslation();

  return (
    <div className="flex gap-2">
      <button type="button" onClick={() => i18n.changeLanguage('fr')}>
        Français
      </button>

      <button type="button" onClick={() => i18n.changeLanguage('ar')}>
        العربية
      </button>
    </div>
  );
}
```

---

## Regeln für Translation Keys

### 1. Keys sind stabil
Ein Key darf nicht ständig umbenannt werden.

### 2. Keys sind logisch gruppiert
Beispiel:
- `patient.form.firstName`
- `report.actions.validate`
- `portal.reports.title`

### 3. Keine Sätze als Key
Nicht:
```text
"ClickHereToCreatePatientNow"
```

Sondern:
```text
patient.actions.create
```

### 4. Gleiche Struktur in FR und AR
Wenn ein Key in `fr` existiert, muss derselbe Key in `ar` vorhanden sein.

## Backend Bezug zu i18n

Das Backend speichert grundsätzlich neutrale Codes und Felder.

Beispiele:
- `testCode = GLU`
- `status = PENDING`
- `flag = H`

Das Frontend übersetzt diese Codes in die sichtbare Sprache.

Ausnahme:
- Report-Kommentare können getrennt in `commentsFr` und `commentsAr` gespeichert werden.

---

## Regeln für Report-Texte

### Kommentare
Reports sollen getrennte Felder besitzen:

- `commentsFr`
- `commentsAr`

### Warum
Der Laborarzt kann einen medizinischen Kommentar gezielt in beiden Sprachen pflegen.

### Regel
- Wenn nur `commentsFr` vorhanden ist, soll das Frontend optional FR anzeigen.
- Wenn `commentsAr` vorhanden ist und die Sprache `ar` ist, soll Arabisch angezeigt werden.

---

## Regeln für Zahlen und Einheiten

- Zahlen selbst werden nicht übersetzt.
- Die visuelle Darstellung von Einheiten kann pro Sprache erfolgen.
- Messwerte bleiben fachlich identisch.

Beispiel:
- Wert: `126`
- Einheit FR: `mg/dL`
- Einheit AR: `ملغ/دل`

---

## Regeln für RTL Layout

Wenn Sprache `ar` ist:

- gesamte App = `dir="rtl"`
- Sidebar rechts oder gespiegelt
- Textausrichtung standardmäßig rechts
- Icons mit Pfeilrichtung prüfen
- Tabellen optisch testen
- Form-Labels und Inputs korrekt ausrichten

Wenn Sprache `fr` ist:

- gesamte App = `dir="ltr"`
- Standardlayout linksbündig

---

## Komponenten mit RTL Sensibilität

Besonders prüfen:

- Sidebar
- Breadcrumbs
- DataTable
- Search Inputs
- Dropdown Menus
- Pagination
- Modals
- Date Picker
- Badge Reihenfolgen
- Icon Buttons mit Pfeilen

## Verbotene Muster

### 1. Harte Texte
Nicht:
```tsx
<button>Créer</button>
```

Richtig:
```tsx
<button>{t('common:actions.create')}</button>
```

### 2. Gemischte Sprachen im Code
Nicht:
```tsx
<label>Nom / الاسم</label>
```

Richtig:
Key-basiert mit einer aktiven Sprache.

### 3. Unterschiedliche Keys in FR und AR
Nicht erlaubt.

### 4. Texte in Utility Dateien
Texte gehören in JSON-Dateien oder zentrale i18n-Konfiguration.

Jeder neue Key muss prüfen:

1. existiert in `fr`
2. existiert in `ar`
3. ist fachlich korrekt
4. ist im richtigen Namespace
5. hat keine Tippfehler
6. funktioniert im UI mit RTL/LTR
